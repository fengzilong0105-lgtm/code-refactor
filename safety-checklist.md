# 安全检查清单与问题处理预案

## 前期检查（动手前必须全部完成）

- [ ] 工作区干净：`git status` 无未提交改动，从稳定基线拉出 `refactor/xxx` 分支
- [ ] 已确认项目的编译/构建命令并成功跑通一次（如 `mvn compile` / `mvn clean install`），作为基线
- [ ] 已确认哪些模块依赖被重构模块（改公共类时需一并编译下游模块）
- [ ] 已生成《重构清单》并与用户确认范围、优先级、批次
- [ ] 已整理对外接口基线清单（URL + 方法 + 入参 + 出参类型）
- [ ] 已评估测试现状；核心类无测试的，先补特征测试固化当前行为
- [ ] 已识别高危区（分库分表、MQ topic、对外推送结构）并标注「默认不碰」

## 中期检查（每个重构单元）

- [ ] 本单元只做一种重构
- [ ] 删除/移动前已全局搜索引用（Java + XML + yml/properties + 反射字符串）
- [ ] 编译通过
- [ ] 相关测试通过
- [ ] 对外接口签名与基线一致
- [ ] 已 commit，message 格式 `refactor(模块): 做了什么`
- [ ] 新发现的 bug / 死代码已登记清单，未顺手修改

## 后期检查（收尾）

- [ ] 全量构建通过（含下游依赖模块）
- [ ] 全量测试通过
- [ ] 接口清单与基线 diff 为空（或差异均已获用户确认）
- [ ] 死代码清单已逐项向用户提问并按确认结果处理，删除后再次编译验证
- [ ] README / 接口文档已更新
- [ ] 规范已沉淀为项目规则（`.cursor/rules/`）
- [ ] 新踩的坑已回写 smells-playbook.md

## 问题处理预案

总原则：**5 分钟内修不好就回滚该单元（`git checkout` / `git revert`），缩小步子重做。**

| 问题 | 处理方式 |
|------|---------|
| 编译失败 | 立即修复；修不好则回滚本单元，把单元拆得更小重做 |
| 测试失败 | 先判断：测试固化了 bug（更新测试并登记），还是重构改坏了行为（回滚） |
| 接口出入参变化 | 红线违规，回滚。确需变更接口时单独立项，征得用户确认并通知调用方 |
| 移动类后运行时报 ClassNotFound / Bean 找不到 | 检查组件扫描范围、XML 中的全限定类名、配置文件中的类名字符串 |
| 拆类后出现循环依赖 | 拆分边界不对：引入第三个类承载共享逻辑，或重新划界。禁止用 `@Lazy` 掩盖 |
| 配置外置后某环境启动失败 | 检查是否漏改某个 profile；配置类加 `@Validated` + `@NotBlank` 启动校验 |
| 删了「死代码」结果有隐藏引用 | `git revert` 恢复；把该引用方式（反射/XML/配置/外部调用）补进前期检查清单 |
| 分库分表相关 SQL 异常 | 回滚本单元；该 Mapper 列入高危区，后续仅做移动不做改写 |
| 重构期间与并行开发分支冲突 | 每 1-2 天 rebase 主干；冲突大的单元延后；必要时与团队约定冻结对应文件 |
| 重构跨度大导致上下文丢失 | 依靠 todo 清单 + 每单元一 commit，任何时刻可回到最近的干净 commit 重新出发 |
