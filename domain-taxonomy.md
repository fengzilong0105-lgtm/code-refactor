# 业务域分类与模块治理

阶段一摸底时，为**当前项目**识别顶层包/模块并标注类别，决定目录结构、DTO 归属与重构顺序。本文件定义**通用分类模型**，不绑定任何具体项目或行业。

标准包结构见 [conventions.md](conventions.md)。

---

## 一、分类模型（A / B / C / D）

按**职责与边界**划分，与包名、行业无关。同一项目内每个顶层业务包只能归为一类。

| 类别 | 名称 | 本质职责 | 识别信号（满足多数即可） |
|------|------|----------|--------------------------|
| **A** | 核心域（Core Domain） | 单一业务能力，拥有或强关联专属数据 | 有独立表/实体；有专属 API 或消息消费；业务名词与包名一致；可独立向产品经理解释「这块做什么」 |
| **B** | 聚合/编排域（Aggregation / Facade） | 跨多域读模型、统计、大屏、组合查询 | 主要调用其他域 Service/Mapper；自身少写或不写核心规则；DTO 多为多源拼装；包名常含 report/dashboard/analytics/facade/portal 等 |
| **C** | 集成域（Integration） | 对外系统对接、出站调用、协议适配 | HTTP/RPC 调外部；Webhook 回调；第三方 SDK 封装；包名常含 integration/adapter/client/gateway/sync/outbound 等 |
| **D** | 基础设施（Infrastructure） | 与具体业务无关的技术能力 | 全局 config、通用 util、启动类、框架扩展、中间件封装 |

### A 类：核心域

**特征**：对应一个有界上下文（Bounded Context），是业务能力的「所有权」所在。

**典型形态（仅作理解，非固定命名）**：

- 电商：`order`、`user`、`product`、`payment`
- 物流：`shipment`、`warehouse`
- 物联网/交通：按**协议类型**或**设备类型**划分的独立模块（如某类数据采集、某类设备管理）
- 后台：`auth`、`role`、`audit`

**结构要求**：

- 具备完整分层：`controller`/`handler`、`service`/`impl`、`dto`/`vo`、`entity`（有持久化时）
- **禁止**将本域专属 DTO/Entity 长期存放在 B 类或 D 类包下

### B 类：聚合/编排域

**特征**：为展示、报表、运营大屏、综合查询服务；**编排**多个 A 类域，而非替代它们承载核心规则。

**典型形态**：

- `report`、`dashboard`、`analytics`、`overview`、`facade`、`portal`、`statistics`

**结构要求**：

- 允许独立顶层包
- `{聚合域}/dto` **仅允许**存放「跨多域拼装」的展示/查询对象（如首页概览、多维报表行）
- **单一来源**的对象（类名、字段、注释能明确归属某一 A 类域）**必须迁回**该 A 类域的 `dto`/`vo`
- `{聚合域}/service` 只做编排与聚合查询，禁止堆积其他域的 entity 业务逻辑
- 过大的 B 类 Service（如 >400 行、import 来自多个域）优先按子场景拆分（批次 7）

### C 类：集成域

**特征**：系统边界之外；出站调用、入站适配、与第三方协议转换。

**典型形态**：

- `integration`、`adapter`、`client`、`gateway`、`sync`、`notify`、`webhook`

**结构要求**：

- 推荐：`{base}.integration.{场景}/service/impl/config/dto`
- 或保留项目既有顶层包名，但必须有 `config/` + `@ConfigurationProperties`
- **禁止**在 Service/Util 中用 `static final` 硬编码 URL、IP、token、密钥
- 不含某 A 类域专属的 CRUD；若出站逻辑仅服务单一核心域，应迁入该 A 类域的 `service` + `config`

**反例（任何项目均须改造）**：

```java
@Service
public class XxxOutboundService {
    private static final String HOST = "10.x.x.x:8088";  // 禁止
    private static final String TOKEN = "xxx";           // 禁止
}
```

### D 类：基础设施

**特征**：被全项目依赖，不表达单一业务概念。

**典型形态**：

- `config`、`common`、`util`、`framework`、`shared`、应用主类所在包

**结构要求**：

- 不放业务 DTO/Entity（共享契约放 `common/dto` 须有明确跨域理由）
- `util/` 禁止直接发起 HTTP 调用（归 C 类）
- 跨域常量、枚举放 `common/constant/` 或 `config/`

## 二、阶段一：如何为任意项目划定业务域

不预设包名。按以下步骤产出《业务域地图》，写入 `REFACTOR_PROGRESS.md`。

### 步骤 1：列出候选顶层包

- Java：列出 `src/main/java/{base}/` 下的一级子包
- 多模块：列出各 module 及其主包
- 排除：测试包、生成代码目录、`target`/`build`

### 步骤 2：用决策树分类

对每个候选包依次判断：

```
是否主要对接外部系统（HTTP/RPC/第三方 SDK）？
  └─ 是 → C 类

是否主要编排多个其他域、自身几乎无核心实体？
  └─ 是 → B 类

是否全局配置/工具/框架、无业务语义？
  └─ 是 → D 类

是否拥有明确业务能力与（通常）专属持久化？
  └─ 是 → A 类

无法判断 → 标为「待确认」，阶段一结束时用 AskQuestion 与用户确认
```

### 步骤 3：标注域间关系

```markdown
## 业务域地图（示例结构，包名按实际项目填写）

| 包/模块 | 类别 | 业务说明 | 依赖的其他域 |
|---------|------|----------|--------------|
| {order} | A | 订单生命周期 | user, product |
| {report} | B | 运营报表聚合 | order, user |
| {payment-gateway} | C | 支付渠道对接 | — |
| {config} | D | 全局配置 | — |
```

### 步骤 4：识别「错放」与「上帝模块」

| 现象 | 判定 | 处理方向 |
|------|------|----------|
| A 类域的 DTO 出现在 B 类 `dto/` | 错放 | 批次 6 迁回来源 A 类域 |
| 单包 `dto/` 文件多且类名前缀对应多个不同 A 类域 | 上帝模块 | B 类瘦身：迁 DTO + 拆 Service |
| C 类包内有 `static` URL 且无 `config/` | 硬编码 | 批次 4 优先 |
| A 类域内 `web` 与 `controller` 并存 | 目录混乱 | 批次 6 统一 |

## 三、错放 DTO / 模型的通用判定

不依赖具体类名前缀，用**归属**判断：

| 信号 | 处理 |
|------|------|
| 类注释、字段名或业务含义仅属于某一 A 类域 | 迁入该域 `dto/` 或 `vo/` |
| 仅被单一 A 类域的 Service/Mapper 引用 | 必须在该域内 |
| 被 3 个以上不同 A 类域引用，且为稳定共享契约 | `common/dto/` 或保留在 B 类并文档注明 |
| 仅为某一聚合页面服务、字段来自多域 | 可留在 B 类 `dto/` |
| 有数据库表映射（`@TableName`、Mapper XML） | 必须在对应 A 类域 `entity/`，不在 `dto/` |

**禁止**用「类名含某缩写」作为唯一规则；以**数据所有权与调用关系**为准。

## 四、上帝模块识别（B / C 类重点）

**识别**（满足其一即列入清单）：

- 单包 `dto/`（或 legacy `po/`）文件数明显偏多，且类型明显分属多个 A 类域
- 单 ServiceImpl 行数 >400，且 import 来自 ≥3 个不同 A 类域
- 包名像业务域，实际承担全站聚合（常见于历史 `report`、`stat`、`admin` 大包）

**处理顺序**：

1. 批次 6：错放模型迁回来源 A 类域
2. 批次 4：C 类硬编码外置（若适用）
3. 批次 7：大 Service 按子场景 Extract Class
4. 批次 5：入口层只调编排 Service，不直调多域 Mapper

---

## 五、重构顺序（与类别挂钩）

在**单域原子完成**策略下，推荐全项目域的**先后顺序**（按风险与依赖，非按包名字母序）：

```
1. D 类（仅配置归类，一般不拆类）
2. C 类（先外置硬编码，再动结构）
3. 体量小、依赖少的 A 类域（练手、验证门禁）
4. 其余 A 类域（按依赖拓扑：被依赖少的先做）
5. B 类聚合域（最后做，避免未迁移 DTO 反复牵动）
```

用户指定范围时，只对该范围内按上表排序。

| 类别 | 域内批次优先序 | 注意 |
|------|----------------|------|
| C | 4 → 6 → 5 | 先配置外置 |
| A | 2 → 4 → 6 → 5 → 8 | 单域做完再做下一域；命名在分层与目录就绪后 |
| B | 6（迁模型）→ 5 → 8 → 7 | 避免半成品扩散 |
| D | 4（配置归类） | 一般不拆类 |
