## 三、三个子任务（单单元只做其一）

每个子任务独立成单元，编译 + 门禁后再做下一子任务。

### 9.1 公共代码提取优化

**识别信号**：

- 同一域内 ≥2 个 Service/Handler 出现相同私有逻辑（桩号规范化、设备在线统计、DTO 拼装骨架）
- 多个类重复 `stream().filter(...).count()` 且条件相同
- 本可放入 `constant`/`util` 的片段仍散落在方法体内

**处理顺序**（由低到高侵入）：

1. **Extract Method**（同类 private → package-private，谨慎 public）
2. **域内 util** → `{域}/util/XxxHelper.java`（仅本域用）
3. **跨域 common** → `common/util/` 或 `core` 模块（须确认 ≥2 个域引用且无循环依赖）
4. **校验/转换** → 优先 Bean Validation、`Converter` 接口，而非巨型 util

**放置规则**：

| 复用范围 | 目标位置 |
|----------|----------|
| 单 Service 内 | private 方法 |
| 单业务域 | `{域}/util/` 或 `{域}/service/support/` |
| 多域、无状态纯函数 | `common/util/` |
| 需 Spring 注入 | `{域}/service/support/` 或独立 `@Component`，禁止 static 持有 Bean |

**禁止**：为抽 3 行代码新建「上帝 Util」；跨域抽取前未查循环依赖。

---

### 9.2 拆分复杂代码以利复用

**识别信号**：

- 单方法 **>50 行**（或圈复杂度明显高：多层 if/for 嵌套 ≥3）
- 一个方法混合「查询 + 转换 + 聚合 + 副作用」（发 MQ、写 Redis）
- 批 7 拆类后，单个方法仍承担完整子流程

**处理手法**：

1. **Extract Method**：按步骤拆为 `loadXxx` / `buildXxx` / `applyXxx`（命名遵循 conventions §6）
2. **Extract Class**：同一 Service 内私有内部类或 `{Domain}XxxSupport`（仍属批 9，不是批 7 的跨文件拆类时可先 internal）
3. **引入策略/模板**：仅当 ≥2 处流程骨架相同、仅细节不同时

**通过标准**：目标方法行数较单元基线下降；主流程方法读起来像「步骤列表」。

---

### 9.3 优化循环与判断（控制流）

**识别信号**：

| 模式 | 示例 |
|------|------|
| 深嵌套 | `for` 内多层 `if`，可 Guard Clause / `continue` |
| 重复条件 | 循环体内多次 `equals` 同一字段 |
| 索引/查找重复 | 循环内 `list.stream().filter` 导致 O(n²) |
| 布尔泥团 | 连续 4+ 个 `&&`/`\\|\\|`，无语义命名 |
| 重复遍历 | 同一集合多次 stream 做可合并的统计 |

**允许（行为保持）**：

- 提前 `return` / `continue` 减少嵌套
- 提取 `boolean isXxx(实体)` 谓词方法
- 合并对同一集合的单次遍历（`for` 一次累加多个计数）
- `Map` 预分组替代内层循环查找（**须确认 key 冲突与顺序语义不变**）

**禁止（须用户单独确认，不得归入批 9）**：

- 改变排序、去重、空值处理语义
- 并行流 `parallelStream`（可能改变顺序与线程安全）
- 算法级优化（如改分页 SQL 为内存过滤）若可能影响结果集

**登记**：无法确认等价性的「性能优化」→ 写入《重构清单》性能待办，不在本批执行。
